# Task 1 Notes

I began by reviewing the EpicBook repository to identify the actual runtime components before introducing any infrastructure or container changes. That review showed the app is a Node.js and Express application using Handlebars, Sequelize, and MySQL, so I defined a clean three-service deployment boundary: application, database, and reverse proxy. This was important because the repository contained traces of older non-Docker deployment ideas, and I needed to stay aligned with the assignment objective rather than inherit an outdated architecture.

From there, I implemented the containerization work in an evolutionary way. I first created a single-stage Docker build as the baseline version, then introduced a multi-stage Docker build as the final production-oriented path. The final build remained multi-stage, and I preserved the single-stage version only as a comparison artifact so I could explain the build optimization story clearly. I measured both images and confirmed that the final multi-stage image was smaller than the single-stage comparison image by about 5.09 MB, which gave me concrete numbers to use in both the report and reflection.

After the build stage, I created a local Docker Compose stack with three services: `docker-epicbook-app`, `docker-epicbook-db`, and `docker-epicbook-proxy`. I added environment-based runtime configuration, healthchecks, structured logging, a reverse proxy layer, and database persistence. The local stack was fully validated through Compose health, reverse-proxy access, and the `/healthz` endpoint before I moved to the cloud deployment stage. One important runtime issue surfaced during this phase: the application still assumed a production database URL environment variable path that was not suitable for local Docker. I corrected that by making the Sequelize startup logic fall back cleanly to standard environment-driven database settings.

For the cloud phase, I provisioned a completely separate EC2 instance specifically for this Dockerized EpicBook work so it would not overlap with previous environments. I used Terraform to create the instance and its security group under isolated `docker-epicbook-*` naming. Then I adapted the repository’s existing Ansible approach away from the older split-host design and rebuilt it around a single-host Docker deployment model. The Ansible playbook installs Docker, synchronizes the repository, places the remote environment file, and runs `docker compose up --build -d` on the VM.

The final deployment was validated on the new EC2 instance after a few operational corrections. I had to address a terminated earlier target, a variable-loading issue in the Ansible playbook, and the fact that the host package name for Compose was `docker-compose-v2` rather than `docker-compose-plugin`. Once those were fixed, the remote deployment completed successfully. The final stack came up healthy on the VM, and the site was publicly reachable through Nginx at `http://52.91.76.250`, with the remote healthcheck also returning `ok`.

Overall, the task achieved the assignment objective: EpicBook was taken from source review to a Dockerized, Compose-managed, reverse-proxied deployment on AWS, with local validation, remote automation, and measurable build optimization evidence.
